Requirements

Revzeroconf is a wrapper for the cross-platform C library Howl. Howl requires a multicast DNS responder to be running on the system. For each operating system this has different implications:

  • On Mac OS X, a mDNSresponder is built into the OS so no special setup is required. Hurrah for Apple.
  • On MS Windows, install the Howl distribution setup exe from porchdogsoft.com. The installer will put mDNSresponder.exe as a system service, and 3 Howl .dlls will be installed on the system (rendezvous.dll, salt.dll and corby.dll). The Howl version must match the RevZeroconf version, e.g. 0.9 or 0.8.2!
  • On Unix, compile and install Howl and then you should have an mDNSresponder executable to run.

The outlook for Rendezvous appears to be good. Over time it will be easier to support, and more users will have Rendezvous ready systems.

Microsoft will support zeroconf in a future OS release (Longhorn). Apple is expected to release a Rendezvous product for MS Windows in 2004. Linux and FreeBSD have moved towards supporting Howl zeroconf in the base OS.

But until zeroconf and the mDNSresponder becomes universally available, installing Howl will be necessary on Windows and Unix.

Rendezvous/Zeroconf Crash Course

Zeroconf is "Zero configuration networking". Rendezvous is Apple's branding of the Zeroconf standard. Zeroconf provides features for:

  • automatic network interface configuration
  • automatic multicast address allocation
  • name-to-address translation
  • address-to-name translation
  • service browsing **
  • service publishing **

Only the last 2 items ** are relevant for revzeroconf: service browsing or discovery, and service publishing.

Service Browsing

A service type declares what network protocol a service is providing. To browse or "discover" for a zeroconf service, all you need to know is the service type. A service type is of the form: _applicationprotocol._transportprotocol.

some common service types:

service type description
_http._tcp. Hypertext Transfer Protocol
_daap._tcp. Digital Audio Access Protocol
_ftp._tcp. File Transfer Protocol
_ldap._tcp. Lightweight Directory Access Protocol
_kitchen-temperature._udp. Kitchen thermometer broadcast protocol
_random-numbers._udp. LavaRnd

developer.apple.com says regarding service types:

"Currently, transportprotocol must be either tcp or udp . The applicationprotocol must only contain letters, digits, hyphens and underscores. ANA restricts the applicationprotocol to be 14 characters or less. Although mDNSResponder doesn't enforce this maximum length, Apple highly recommends that your applicationprotocolbe 14 characters or less so it can be registered with IANA."

See zeroconf.org and RFC2782 for details on the format of service type names.

Service Publishing

To publish or announce a service requires a service type and 3 other bits of information:

  1. tcp/ip port number
  2. service name
  3. service text

For example iTunes uses the following service definition, on my computer:

part of definition what is it
_daap._tcp. service type
3689 tcp/ip port number
alex itunes service name
_txtvers=1Version=131072 Machine ID=EE0E1AF0771D Database ID=466C683590F917A9 Machine Name=alex itunes Password=false service text

As a developer publishing a service you should be careful to choose a service type and port number that do not conflict with other network services on the computer or server.

Service name and service text, I think, can be arbitrary: use whatever is helpful for you to implement your app.

Service text is expected to be formatted in DNS text record format. According to the Howl docs:

"The string must be in the format specified by DNS text records (e.g. key|key=|key=val). The string must be no longer than 255 octets."

Transcript API

zconfInit
type transcript command
purpose initialize the zeroconf networking library. this command must be called before any other revzeroconf commands.
parameters none
returns empty (is a command not a function)
error condition will set "the result" function to an error message if the zerconf library could not be initialized.
sample code

zconfInit
if the result <> empty then
  answer the result
  exit to top
end if

zconfFinish
type transcript command
purpose stops the previous browse and/or publish command. releases the browse and publish objects. finalizes the zeroconf library. (important **: see Known Issues for important usage notes)
parameters none
returns empty (is a command not a function)
error condition will set "the result" function to an error message if the zerconf library could not be finalized.
sample code

zconfFinish
if the result <> empty then
  answer the result
  exit to top
end if

zconfSetStatusCallback <message name>, <long id of target control>
type transcript command
purpose sets up a callback message so you can get browse or publish events. Do this command before browsing or publishing services so revzeroconf can notify you of network events.
parameters 2, required
returns empty (is a command not a function)
error condition sets "the result" function
sample code

zconfSetStatusCallback \
     "myZeroconfStatus", the long id of this card
if the result <> empty then
  answer the result
  exit to top
end if

on myZeroconfStatus pMessage
  put "zeroconfig message:" && pMessage
end myZeroconfStatus

zconfBrowseServiceType <service type>
type transcript command
purpose starts a browse session for the service type. browse answers will be sent to the handler you set with zconfSetStatusCallback
parameters 1, required
returns empty (is a command not a function)
error condition sets "the result" function
sample code

zconfBrowseServiceType "_http._tcp."
if the result <> empty then
  answer the result
  exit to top
end if

zconfPublishService <name>, <type>, <port>, [service text]
type transcript command
purpose starts a publish session for the service definition. publish confirmation will be sent to the handler you set with zconfSetStatusCallback
parameters 4, 3 required, 1 optional
returns empty (is a command not a function)
error condition sets "the result" function
sample code

zconfPublishService \
  "funny joke service", \
  "_http._tcp.", \
  8081, \
  "connect http on port 8081 for a good one"
if the result <> empty then
  answer the result
  exit to top
end if

zconfBrowseDomains
type transcript command
purpose this command is supposed to setup replies to discover what zeroconf domains are existant. e.g. "local." however, it may not do anything, and is implemented here because Howl has it. maybe it will do something interesting in the future.
parameters none
returns empty (is a command not a function)
error condition sets "the result" function
sample code

zconfBrowseDomains
if the result <> empty then
  answer the result
  exit to top
end if

 

Status Messages

revzeroconf will send messages to your callback handler when network events occur. Use the zeroconfWorkbench stack to experiment with browsing and publishing. Then you will get a feel for what all these messages are.

An important concept is that revzeroconf sometimes sends messages in pairs. First a REPLY message, followed by a RESOLVE message. The REPLY message says what action is happening. The RESOLVE message contains additional information including the IP address of the computer to which the REPLY message corresponds. The RESOLVE message should be immediately following the REPLY message.

Resolve Reply Messages

format:

resolve reply: name,type,domain,IP,port,textRecord

example (note: 1 long line):

resolve reply: alex itunes,_daap._tcp.,local.,0.0.0.0,3689,txtvers=1?Version=131072?Machine ID=EE0E1AF0771D?Database ID=466C683590F917A9?Machine Name=alex itunes?Password=false

Browse Reply Messages

format:

browse reply: instruction
browse reply: instruction,name,type,domain

examples:

browse reply: Add Service,alex itunes,_daap._tcp.,local.
browse reply: Remove Service,alex itunes,_daap._tcp.,local.
browse reply: Invalid
browse reply: Release
browse reply: Add Domain
browse reply: Add Default Domain
browse reply: Remove Domain
browse reply: Resolved

Just ignore the messages that you don't need or understand. I'm not sure what the intended use of some of those messages are, but there they are.

Publish Reply Messages

format:

publish reply: result

examples:

publish reply: Started
publish reply: Stopped
publish reply: Name Collision
publish reply: Invalid

Known Issues

  • The current implementation of revzeroconf is simplistic: it only can handle 1 browse session and 1 publish session at once. When you call zconfFinish, only the previous 1 browse and previous 1 publish sessions are stopped and cleared. For example, this is OK:

    zconfInit
    zconfBrowseServiceType "_http._tcp."
    zconfFinish -- later on
    zconfInit
    zconfBrowseServiceType "_daap._tcp."
    zconfFinish -- later on

    But in the following example, the _daap_.tcp. service type will be orphaned. It will however continue to be active, and continue to send events back to your callback message. However, zconfFinish will not clear it and there is no way to stop the browse of the service type.

    zconfInit
    zconfBrowseServiceType "_http._tcp."
    zconfBrowseServiceType "_daap._tcp." -- we have lost control of the _http._tcp. session
    zconfFinish -- later on; only the _daap._tcp. session

    Again: you will continue to get messages from both browse sessions, http and daap. Which is fine. It's only a problem if you must stop the browse for http, because you can't.

    Note that the same problem applies to publishing of service types (zconfPublishService) and could be a bigger problem in that case.

    But it seems unlikely that one would need to browse and publish multiple service types AND need to start and stop them. After all, your browse and and publish sessions will stop when your app quits.
  • You tell me.
